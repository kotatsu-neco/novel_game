# sn_text_display_spec_v01.md

## 0. 文書の位置づけ

この文書は、サウンドノベルスターター / シナリオ制作キットにおける、次工程 `sn_engine_v02a` / `sn_author_kit_v02a` 向けの**日本語文字表示・文字送り・ページ分割仕様案**である。

現行の `sn_engine_v01` は、固定値ベースのページ分割に寄っている。

```text
text     : 本文
voice    : 台詞
document : 文書
```

現行の固定値例：

```text
本文   : 19文字 × 10行
台詞   : 17文字 × 8行
文書   : 17文字 × 12行
```

これは特定端末では成立し得るが、販売品質を目指す場合、端末幅・端末高さ・フォントサイズ設定・ブラウザ差・日本語禁則処理に耐えにくい。

そのため、次工程では**固定文字数主導ではなく、実表示領域測定と日本語禁則処理を主軸にした文字表示エンジン**へ移行する。

---

## 1. 目的

### 1.1 主目的

`sn_engine_v02a` では、以下を実現する。

```text
端末ごとの画面幅・高さ・文字サイズ設定に合わせて、
本文 / 台詞 / 文書を安全に表示できる文字送り・ページ分割を行う。
```

### 1.2 副目的

```text
1. 句読点だけの行・ページを発生させない。
2. 閉じ括弧だけの行・ページを発生させない。
3. 小書き仮名・長音符などが不自然にページ頭へ孤立しないようにする。
4. 文字サイズ「大」でも表示が破綻しない。
5. [r] 明示改行、[p] 明示改ページを仕様として扱う。
6. CSS任せではなく、エンジン側に禁則判定レイヤーを持つ。
7. シナリオ作成キット側にも、文章量・改ページ・禁則に関する作成ルールを反映する。
```

## 2. 参照基準

### 2.1 主要参照

- W3C: Requirements for Japanese Text Layout / JLREQ
  - 日本語組版の要件、行頭禁則、行末禁則、約物、小書き仮名、行分割等の基礎参照。
- CSS Text Module
  - `line-break` などCJKテキストの行分割制御に関するCSS仕様。
- MDN Web Docs
  - `getBoundingClientRect()`
  - `scrollHeight` / `clientHeight`
  - `ResizeObserver`
  - `requestAnimationFrame()`
  - `CanvasRenderingContext2D.measureText()`
- ノベルゲーム制作系の既存知見
  - Ren'Py のテキスト表示処理の考え方
  - TyranoScript / TyranoBuilder の `[r]`、`[p]`、文字表示速度、クリック待ち等の概念

### 2.2 参照の扱い

このエンジンでは、JLREQを丸ごと完全再実装するのではなく、まず次を**販売品質上必要な実装範囲**として定義する。

```text
1. 横書き・スマホ縦画面・短編サウンドノベルに必要な日本語禁則処理。
2. ページ分割と文字送りに直接影響する禁則処理。
3. 本文 / 台詞 / 文書の表示に必要な最小限ではなく、早期に実用品質へ寄せる禁則処理。
```

縦書き対応は別工程。ただし、禁則処理モジュールは将来の縦書き対応でも再利用できる構造にする。

## 3. 現状の問題整理

### 3.1 確認済み

```text
1. 現行表示は、ユーザー端末の見え方に寄せて文字数・行数を調整している。
2. 台詞や文書が右端・下端で見切れる問題が発生した。
3. 句点・読点だけが次ページに送られる問題が発生した。
4. 文字サイズ設定を導入したため、ページ分割条件がさらに変動する。
5. 設定UIでは、本文 / 台詞 / 文書の文字送り速度と文字サイズを独立設定できるようになった。
```

### 3.2 想定

```text
1. iPhone SE系、標準iPhone、大型iPhone、Android、PCブラウザで表示結果が変わる。
2. 文字サイズ「大」では、現行の固定文字数ベース分割が破綻する可能性が高い。
3. 長文の文書表示、手紙、覚書、張り紙は表示破綻リスクが高い。
4. CSSの `line-break` だけではページ分割までは制御できない。
```

### 3.3 未確認

```text
1. 実ブラウザでのDOM測定方式の挙動。
2. iPhone Safariでの測定精度。
3. Android Chromeでの禁則・折り返し差。
4. 文字サイズ「大」時のすべての本文 / 台詞 / 文書の表示安定性。
5. requestAnimationFrame方式の文字送り体感。
```

## 4. 設計原則

### 4.1 固定文字数方式の降格

現行の `charsPerLine / maxLines` は廃止しない。

ただし、役割を変更する。

```text
現行：
  主判定

v02以降：
  DOM測定が失敗した場合のフォールバック
  あるいは静的検査時の概算値
```

### 4.2 実DOM測定を主判定にする

ページ分割の主判定は、非表示の測定用DOMを使って行う。

```text
1. 実際の表示領域と同じ幅・フォント・行間・CSSを持つ測定用DOMを用意する。
2. 候補テキストを流し込む。
3. scrollHeight <= clientHeight なら収まると判定する。
4. 超える場合は、禁則処理を踏まえて安全な分割位置まで戻す。
```

### 4.3 文字送りとページ分割を分離する

文字送りは、ページ分割後の**1ページ分テキスト**に対して行う。

```text
原文
↓
手動改ページ / 手動改行の解釈
↓
禁則処理つきページ分割
↓
ページ配列
↓
各ページに対して文字送り
```

### 4.4 速度は経過時間ベースにする

文字送りは `requestAnimationFrame()` を使い、フレーム数ではなく経過時間で表示文字数を決める。

```text
表示文字数 = floor((現在時刻 - 開始時刻) / msPerChar)
```

## 5. 新規モジュール設計

### 5.1 追加予定ファイル

```text
src/engine/japaneseLayoutRules.js
src/engine/textPaginator.js
src/engine/textMeasure.js
src/engine/typewriterController.js
```

### 5.2 japaneseLayoutRules.js

役割：

```text
1. 文字クラスを定義する。
2. 行頭禁止文字を判定する。
3. 行末禁止文字を判定する。
4. ページ頭禁止文字を判定する。
5. ページ末禁止文字を判定する。
6. 孤立禁止パターンを判定する。
7. 分割候補を補正する。
```

### 5.3 textMeasure.js

役割：

```text
1. 測定用DOMを生成する。
2. 本文 / 台詞 / 文書ごとのCSSを反映する。
3. 現在の表示領域サイズを取得する。
4. 候補テキストが収まるか判定する。
5. サイズ変更時に測定キャッシュを破棄する。
```

### 5.4 textPaginator.js

役割：

```text
1. [p] 明示改ページを最優先で処理する。
2. [r] 明示改行を保持する。
3. 段落単位・文単位・読点単位・文字単位の順で分割候補を作る。
4. 測定結果に基づきページを作る。
5. 禁則違反を検出し、前後のページへ文字を移動して補正する。
6. それでも不可能な場合はfallback値に基づき安全側に分割する。
```

### 5.5 typewriterController.js

役割：

```text
1. requestAnimationFrameベースで文字送りする。
2. msPerCharに基づき表示文字数を決定する。
3. タップ1回目で全文表示する。
4. 全文表示後のタップで次ページへ進む。
5. ページ最終行・最終文字・選択肢表示との衝突を防ぐ。
```

## 6. 禁則処理仕様

### 6.1 行頭禁止文字

初期実装で扱う行頭禁止文字は以下。

```text
、 。 ， ． , . ！ ？ ! ? 」 』 ） 】 〕 〉 》 ぁ ぃ ぅ ぇ ぉ ゃ ゅ ょ っ ー
```

検討対象として追加候補：

```text
ァ ィ ゥ ェ ォ ャ ュ ョ ッ ヵ ヶ
ゝ ゞ ヽ ヾ
々
```

### 6.2 行末禁止文字

初期実装で扱う行末禁止文字は以下。

```text
「 『 （ 【 〔 〈 《
```

### 6.3 ページ頭禁止

ページ頭には以下を置かない。

```text
1. 句読点だけ
2. 閉じ括弧だけ
3. 行頭禁止文字で始まる短い断片
4. 小書き仮名だけ
5. 長音符だけ
6. 三点リーダーの片割れだけ
```

### 6.4 ページ末禁止

ページ末には以下を極力置かない。

```text
1. 開き括弧だけ
2. 開き括弧の直後で終了
3. 次ページ頭が句読点・閉じ括弧になる分割
```

### 6.5 孤立禁止

以下のようなページ・行を禁止する。

```text
「、」
「。」
「？」
「！」
「」」
「』」
「ー」
「っ」
「ゃ」
```

### 6.6 分割優先順位

```text
第1優先：
  [p] 明示改ページ

第2優先：
  段落区切り

第3優先：
  句点・感嘆符・疑問符の後

第4優先：
  読点の後

第5優先：
  自然な短いまとまり

第6優先：
  文字単位

最後：
  禁則違反を最小化する強制分割
```

## 7. 手動タグ仕様

### 7.1 [p] 明示改ページ

```text
[p]
```

意味：

```text
その位置でページを分ける。
[p] 自体は表示しない。
[p] の直後が禁則違反になる場合、エンジンは直前ページから文字を移動して補正する。
```

### 7.2 [r] 明示改行

```text
[r]
```

意味：

```text
その位置で改行する。
[r] 自体は表示しない。
```

### 7.3 将来候補

```text
[w=500]
```

意味候補：

```text
文字送り中に500ms待つ。
v02では未実装でよい。
```

## 8. CSSとの役割分担

### 8.1 CSSで指定するもの

```css
line-break: strict;
word-break: normal;
overflow-wrap: anywhere; /* emergency fallback */
```

### 8.2 エンジンで制御するもの

```text
1. ページ分割位置。
2. ページ頭・ページ末の禁則。
3. 句読点・閉じ括弧だけの孤立防止。
4. 手動改ページ [p] との整合。
5. 文字サイズ変更時の再ページ分割。
```

### 8.3 CSSのみでは不可とする理由

CSSはブラウザの行分割に効くが、サウンドノベルのページ単位表示では、どこまでを1ページとして表示するかをエンジンが決定する必要がある。

### 9.1 監視対象

```text
2. stage領域
3. 設定変更によるhtml data属性
4. orientation change
5. viewport resize
```

### 9.2 挙動

```text
1. 表示領域サイズが変わる。
2. 現在stepのページ分割キャッシュを破棄する。
3. 現在のstepを再ページ分割する。
4. 現在ページ番号が範囲外なら、最後のページへ補正する。
5. 文字送り中なら、安全に停止または再開始する。
```

## 10. 文字サイズ設定との連動

### 10.1 対象

```text
本文
台詞
文書
```

### 10.2 必須要件

文字サイズ変更時は、次を行う。

```text
1. CSS data属性を更新する。
2. 測定用DOMにも同じ設定を反映する。
3. ページ分割キャッシュを破棄する。
4. 現在表示中テキストを再ページ分割する。
5. 表示崩れが出ないことを確認する。
```

## 14. シナリオ作成キットへの反映

### 14.1 作者向けルール

```text
1. 長い文書は [p] で意図的に分ける。
2. 台詞は1発話を長くしすぎない。
3. 本文は説明を詰め込まず、読む単位を意識する。
4. [r] は改行、[p] は改ページとして扱う。
5. [p] 直後に句読点・閉じ括弧だけが来るような書き方を避ける。
6. 文字サイズ「大」でも読めることを確認する。
```

### 14.2 AI向けルール

```text
1. SCENARIO_SOURCE.md 生成時、長い文書には [p] を入れる。
2. 1つの voice step を長くしすぎない。
3. 句読点だけでページが進む可能性のある文を避ける。
4. 文書stepは段落間の空行を使いすぎない。
5. 表示確認対象として「文字サイズ大」を必ず含める。
```

## 18. 役割担当レビュー

### 正本確認担当

この仕様は、次工程の `sn_engine_v02a` / `sn_author_kit_v02a` の正本候補として扱える。実装時は、リポジトリ内の `SPEC.md`、`ARCHITECTURE.md`、`AGENTS.md`、`REPORT.md` へ統合する。

### UI/UX担当

文字が読めない、見切れる、句読点だけが送られる、文字サイズ変更で崩れる、これらは販売品質上の重大問題。文字表示エンジン汎用化は機能追加より優先度が高い。

### アクセシビリティ担当

文字サイズ変更を導入した以上、ページ分割も連動しなければならない。文字サイズ「大」を必須検査対象にする。

### 実装担当

DOM測定ベースのページ分割を主判定にし、`charsPerLine / maxLines` はフォールバックへ降格する。禁則処理は `japaneseLayoutRules.js` として分離する。

### 検証担当

禁則テスト用の固定シナリオを必ず用意する。実DOM測定の正否は、チャットではなくCodex / ローカル / 実機検証に回す。

### 批評・矛盾検出担当

「完全準拠」という語は曖昧に使わない。v02では「JLREQを参照した、横書きサウンドノベル用の実用的な禁則処理」として受け入れ範囲を明記する。

### 統合ディレクター

次工程は `sn_engine_v02a` と `sn_author_kit_v02a`。主題は**日本語文字表示エンジン汎用化**で確定。縦書きより先に実施する。
